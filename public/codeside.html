Diagnosis — AuthFlow SaaS · Security & Backend Review
Reviewed by Dan K. · Engineering + Security · 2.5 days
3 findings
CRITICAL

JWT secret hardcoded in source — exposed in repo history

Your JWT signing secret is committed in plain text. Anyone with access to the repository — or its history — can forge authentication tokens for any user. Rotate this immediately, even before fixing the code.

src/lib/auth.ts:14 · config/env.example · git log --all -S "JWT_SECRET"
HIGH

Missing authorisation check on /api/users/:id — IDOR

The user profile endpoint validates that the caller is authenticated, but not that they own the resource they're requesting. Any logged-in user can read any other user's profile by iterating numeric IDs.

src/api/users/[id].ts:38 — add: if (session.userId !== params.id) return 403
MEDIUM

No rate limiting on auth endpoints — brute-force trivial

Login and password-reset endpoints have no rate limiting. A brute-force attack on a 6-digit OTP would succeed in under a minute. Add exponential backoff or a simple IP-based limiter before this goes anywhere near real users.

src/api/auth/login.ts · src/api/auth/reset.ts — consider: upstash/ratelimit or express-rate-limit
DK

"Two ship-blockers. Fix the hardcoded secret and the IDOR today — they're not optional. The rate limiting matters but can follow in the next sprint. Everything else I saw was tidy. This app has good bones; just patch the holes."

— Dan Kowalski, Application Security · 13 yrs
